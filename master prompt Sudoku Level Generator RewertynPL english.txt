BEWARE OF THESE PARADOXES:
A) The ultimate Catch-22 in generating Level 8 (Theoretical) puzzles from extremely opposite sides of the barricade (Generator vs. Certifier).

Side A: The Certifier's Perspective (Why I advised a SPARSE grid):

Problem: Naked Single eats your board.

What is happening? The grid is so dense (has so many clues) that the simplest Level 1 function (apply_naked_single) solves the entire Sudoku with its eyes closed. The engine enters the Certifier, sees a dense board, fires P1, gets basic.solved = true and exits. The code for P8 (POM / MSLS) is never even called because there is no bottleneck.

Side B: The Generator's Computational Perspective (Why the other window advised a DENSE grid):

Problem: Combinatorial Explosion (NP-Hard Complexity).

What is happening? Strategies like the Pattern Overlay Method (POM) rely on overlaying all possible legal placements for a given digit. If the grid is sparse (e.g., 20 clues on a 9x9), the number of legal overlays for one digit runs into the hundreds of thousands. If you run POM on this, your CPU will choke, RAM will run out, and the probe_budget and pattern_cap limits (which are hardcoded in your code) will be exhausted instantly, returning ApplyResult::NoProgress (logical timeout).

How to solve this Paradox? (Architectural Solution):
You are at a dead end:

Grid is too sparse 👉 POM/MSLS explodes due to computational limits and returns NoProgress.

Grid is too dense 👉 Naked Single solves everything immediately and returns basic.solved, bypassing POM/MSLS.

The solution in HPC architectures is not to choose one of the extremes, but The Goldilocks Zone of density combined with strict structure protection.

Modify the "Clue Ceiling" to an intermediate value (approx. 35% - 48%). You cannot use 70-80% (because of Naked Singles), nor 22% (because the graph will explode).

The Digger MCTS must protect the "Blind Spots" (Dead Zones).

OR:
The clue window is currently too "averaged out," which ends up hurting both sides of the fence.

In resolve_auto_clue_range(), you have a single, shared mechanism for both the generator and the smoke/certifier, with bolted-on exceptions per pattern family. For many difficult families, you are actually raising the lower bound of the clue density (std::max(min_clues, ...)), effectively pushing the boards toward denser states. This helps preserve the structure, but at the same time, it frequently increases the chance of early simple solutions or alternative certifications. On the flip side, for patterns like Sue de Coq or Death Blossom, you deliberately use a sparser window. So the code itself already proves that a single "golden mean" just doesn't cut it.

My assessment:
Yes, there should be 2 separate clue policy frameworks:
generator window: serves to keep the pattern alive and protect the core,
certifier/smoke window: serves the purpose of honest validation, not ease of generation.
A single, averaged window performs poorly because it tries to optimize for two conflicting goals at once.



B) The "Self-Destructing Patterns" Problem
This is the second of the two hardest problems in reverse Sudoku generation engines (MCTS + DLX). You have collided with the problem of "Self-Destructing Patterns" in the Generator.
Practically 100% of the board was solved by the simplest strategy in the world—Naked Single (Level 1)! The board never "got stuck", so the certifier NEVER reached Level 6, where your ALS-XZ is located. Since the certifier never got there, the function wasn't called at all, hence use: 0.

C) Strategy Overshadowing
In advanced Sudoku engines, this is called "Strategy Overshadowing". The problem lies in the execution order of the strategies.
Mathematically, ALS-XZ is nothing more than a specific type of chain (often equivalent to an XY-Chain or a short AIC). In your code, at Level 6, the very generic and powerful X-Chain and XY-Chain are called BEFORE ALS-XZ.
When the solver encounters the ALS-XZ pattern injected by the generator, the XY-Chain immediately recognizes it as a bivalue chain, makes eliminations, and returns Progress. The certifier loop starts anew, and the apply_als_xz function is never even called (hence use: 0). The certifier states the puzzle was solved using XY-Chain, and the ALS-XZ contract is rejected.
How to fix it?
For the algorithms to work correctly, we must always run "Named Structures" (Specific Templates) before "General Graph Chains" (Generic Templates). We must swap the order in sudoku_logic_engine.h for Levels 6 and 7.

📝 Summary: What is in each component?
Logic Engine (logic/ folder)

Architectural Rule: The GenericLogicCertify class in sudoku_logic_engine.h is just a dispatcher. It contains a massive while (board.empty_cells != 0) loop and calls static functions like apply_x_wing(st, stats, result) sequentially, injecting a reference to CandidateState (the so-called st).

Zero-Allocation: Strategy files (*.h) are not heap-instantiated classes. They are namespaces (namespace sudoku_hpc::logic::p4) or static inline functions. If a strategy needs a list of cells (e.g., finding nodes for Remote Pairs), it queries ExactPatternScratchpad& sp = exact_pattern_scratchpad(); from thread_local memory, which resets respective C-style arrays (sp.node_count = 0;).

Flat Structures: Arrays like bfs_queue and als_list are preallocated to the maximum size for a 64x64 grid (MAX_NN = 4096).

Module 1: Pattern Forcing (generator/pattern_forcing/)

This is a game-changer for NP-hard Level 7 puzzles.

Instead of blindly removing digits from a finished board, template_exocet.h "seeds" a mathematical structure (e.g., picks base a, b and targets x, y, forcing common candidate masks for them). Then dlx_solver.h steps in, treats these cells as "protected" with limited candidates, and fills the rest of the board with a valid solution in a fraction of a second. The Digger later cannot uncover the "protected" cells.

This ensures the generated board has a guaranteed, built-in P8 pattern.

Module 2: MCTS Bottleneck Digger (generator/mcts_digger/)

Standard depth-first / random search is too slow for generating "Nightmare/Theoretical".

bottleneck_digger.h treats the digit removal process like a game (like in AlphaGo). The game state is the current board with holes.

The action is removing a pair/cell.

Reward: If GenericLogicCertifyP1P5Modular (a solver with limited intellect) gets bogged down (can't solve it), but full DLX shows 1 unique solution, the node gets a massive reward. The MCTS tree guides the digger (UCB1 = exploit + explore) to pull digits that create logical bottlenecks.

Threads and Concurrency (concurrency/persistent_thread_pool.h)

Implements an "Extreme Low-Latency Thread Pool". Fixed bugs:

Worker threads wait on std::atomic<uint64_t>::wait.

The seen_epoch semaphore is updated before waking, preventing missed wake-ups.

Thread reseeding updates the PRNG state in isolation, preventing generation collisions in the pool.
